In the [previous exercise][previous-lesson] you learned how custom instructions give Copilot the project context it needs to generate code that matches your team's standards. Now you'll see that play out in your editor: send a prompt with the *current* instructions, then update **`.github/copilot-instructions.md`** and watch the output change.

In this exercise, you will:

- Send a code-generation prompt to Copilot Chat in VS Code.
- Add a new repository-wide standard to **`.github/copilot-instructions.md`**.
- Re-run the same prompt and observe how the generated code changes.

<Aside type="caution">
  GitHub Copilot is non-deterministic — the exact code it generates may vary between runs. The goal here is to see the *trend* in behavior change after updating the instructions, not to match the output character-for-character.
</Aside>

## Before you begin

You'll be making code changes, so create a branch to work in.

1. Return to your codespace from the previous exercise.
2. Open a new terminal by selecting <kbd>Ctrl</kbd>+<kbd>`</kbd>.
3. Create and switch to a new branch:

   ```bash
   git checkout -b custom-instructions-vscode
   ```

## Use Copilot Chat *before* updating the instructions

To see the impact of custom instructions, you'll start by sending a prompt with the current instruction file in place. Later, you'll update it and re-send the same prompt.

1. Close any open editor tabs from previous exercises so Copilot picks up only the context you want it to have.
2. Open `server/routes/publishers.py` (an empty file).
3. Open Copilot Chat by selecting the Copilot icon at the top of the codespace if it isn't already open.
4. Start a fresh conversation by typing `/clear` in the chat input and pressing <kbd>Enter</kbd>.
5. Select **Ask** from the chat mode dropdown.
6. Send the following prompt:

   ```plaintext
   Create a new endpoint to return a list of all publishers. It should return the name and id for all publishers.
   ```

7. Copilot explores the project and proposes code, often spanning `publishers.py`, `app.py`, and tests.
8. Notice the generated function signatures use [type hints][python-type-hints] — that's coming from a directive already in **`.github/copilot-instructions.md`**.
9. Notice the generated code **is missing** docstrings and a file-level comment header.

<Aside type="caution">
  Because Copilot is probabilistic, there's a chance it'll add docstrings even without being told to. If that happens, that's fine — proceed with the exercise; the *consistency* improvement after the instruction update is still the point.
</Aside>

## Re-run the prompt and observe the change

1. Return to Copilot Chat and select **New Chat** to clear the buffer.
2. Click back into **`server/routes/publishers.py`** so Copilot focuses on the right file.
3. Send the **same prompt** as before:

   ```plaintext
   Create a new endpoint to return a list of all publishers. It should return the name and id for all publishers.
   ```

4. Notice that the newly generated file now opens with a comment block similar to:

   ```python
   """
   Publisher API routes for the Tailspin Toys Crowd Funding platform.
   This module provides endpoints to retrieve publisher information.
   """
   ```

5. Notice that the generated function now includes a docstring similar to:

   ```python
   """
   Returns a list of all publishers with their id and name.

   Returns:
       Response: JSON response containing an array of publisher objects
   """
   ```

6. **Don't accept the changes yet** — you'll generate publishers code in a later exercise. If you accidentally accepted them, use **Undo** at the top of the Copilot Chat panel.

You just steered Copilot to follow a new project standard with a single line of markdown — no code changes, no tool configuration. That's the power of repository custom instructions.
