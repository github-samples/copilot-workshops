# Authoring Guide

This is the entry point for **content authors and maintainers** of the *Agents in SDLC* workshop. If you arrived here looking to *take* the workshop, head to [`workshop/README.md`](./workshop/README.md) or the published site at <https://github-samples.github.io/agents-in-sdlc/>.

## Mental model

- **You only ever edit MDX.** All authored content lives under `docs/src/content/docs/`. That's the source of truth.
- **Two consumers render that source.** The Astro + Starlight site in `docs/` builds the published GitHub Pages site. The Python script `scripts/render-markdown.py` produces a plain-Markdown mirror at `workshop/` so people can take the workshop directly on github.com without leaving the repo.
- **Never edit `workshop/`.** It's regenerated by CI on every push to `main` and any manual edits will be overwritten.

```
agents-in-sdlc/
├── workshop/                     ← generated; bot-owned. Do not edit.
├── docs/
│   └── src/content/docs/         ← MDX source. EDIT HERE.
│       ├── cli/                  ← Copilot CLI lessons
│       ├── vscode/               ← VS Code lessons
│       ├── cloud/                ← Cloud agent lessons
│       ├── shared/               ← Lessons referenced by multiple paths
│       ├── _partials/            ← Reusable MDX fragments
│       ├── images/               ← Screenshots and diagrams
│       └── index.mdx             ← Workshop landing page
├── scripts/
│   └── render-markdown.py        ← MDX → workshop/ Markdown renderer
└── .github/
    ├── copilot-instructions.md   ← AI authoring guide (humans can read it too)
    ├── instructions/             ← Per-file-type conventions (auto-applied to Copilot)
    └── workflows/                ← CI for site build + workshop/ regeneration
```

## Step-by-step recipes

### Add a new lesson

1. **Pick a path and number.** Lessons live under `docs/src/content/docs/{cli,vscode,cloud}/N-name.mdx`. `N` is the next available integer in that path; the number drives the URL slug (`/cli/4-generating-code/`).
2. **Create the file** with frontmatter:
   ```mdx
   ---
   title: "Exercise N - Short descriptive title"
   ---

   import { Aside } from '@astrojs/starlight/components';

   Body starts here.
   ```
   Only `title` is required; it becomes the H1 and the page title. Don't add a body H1 — Starlight renders the title automatically.
3. **Write the body.** Use Markdown plus optional MDX `<Aside>` admonitions. See **Style essentials** below.
4. **Register in the sidebar.** Open `docs/astro.config.mjs` and add an entry to the appropriate `items: []` block. The sidebar is *manually* maintained — order in the file is the order learners see.
5. **Preview locally** (optional but recommended):
   ```bash
   cd docs && npm install && npm run dev
   ```
   Open <http://localhost:4321/agents-in-sdlc/>.
6. **Commit and open a PR.** CI runs the renderer, the Astro build, and lychee against the rendered Markdown. All three must pass.
7. **After merge, the bot regenerates `workshop/`** and commits it back to `main`. You don't need to commit anything under `workshop/` from your branch.

### Add a partial

Partials are reusable MDX fragments imported by lesson pages.

1. **Pick a category prefix.** Partial filenames must start with one of:
   - `callout-` — single short callout (e.g., one `<Aside>`).
   - `section-` — multi-paragraph reusable content (intros, recaps).
   - `exercise-` — self-contained hands-on steps that drop into a numbered list on the host page.
2. **Create the file** at `docs/src/content/docs/_partials/<prefix>-name.mdx`. No frontmatter — the host page's frontmatter applies.
3. **Self-contained links.** Any reference-style link (`[label][ref]`) used inside the partial must define `[ref]: url` *inside the same partial*. Don't depend on the host page to provide refs.
4. **Import and use** in the host page:
   ```mdx
   import StartCli from '../_partials/callout-start-copilot-cli.mdx';

   <StartCli />
   ```
5. **Threshold for extraction**: only extract a partial when the same content appears in 2+ places, or it's at least 5 lines and non-trivial. Single-use content stays inline.

See [`.github/instructions/partials.instructions.md`](./.github/instructions/partials.instructions.md) for the full partial conventions.

### Add an image

1. **Drop the file** in `docs/src/content/docs/images/`. Use lowercase-with-hyphens filenames; prefix with `shared-` if the image is referenced from multiple paths.
2. **Reference it** with a relative path from the consuming MDX page:
   ```mdx
   ![Description of the screenshot](../images/my-screenshot.png)
   ```
3. **Always include alt text.** Both Starlight and the Markdown renderer treat the alt text as required.
4. **Preview locally** to confirm the image resolves.

### Edit an existing lesson

1. **Find the file** under `docs/src/content/docs/` (use the published URL as a hint — `/cli/4-generating-code/` lives at `cli/4-generating-code.mdx`).
2. **Edit the MDX.** Same conventions apply — see **Style essentials** below.
3. **Preview** with `npm run dev` in `docs/`.
4. **Commit, PR, merge.** Same flow as adding a lesson.

### Use a shared lesson across paths

When the same lesson applies to multiple paths (CLI, VS Code, Cloud), put the real content in `docs/src/content/docs/shared/<topic>/<lesson>.mdx` and create thin **redirect-stub** pages in each per-path folder. Stubs include a "Return to your path" footer so learners get back to their track.

See `shared/cloud-agent/cloud-agent.mdx` and the `cloud/2-cloud-agent.mdx` / `vscode/4-cloud-agent.mdx` stubs for the established pattern.

## Local preview

The Astro dev server is the primary preview surface:

```bash
cd docs
npm install
npm run dev
```

The site runs at <http://localhost:4321/agents-in-sdlc/> with hot reload.

If you want to preview the GitHub.com Markdown view (what `workshop/` will look like), run the renderer:

```bash
python scripts/render-markdown.py
```

Output lands in `workshop/`. **Do not commit changes to `workshop/`** — the bot owns it. Running this locally is purely for preview.

## Style essentials

A short cheat sheet. For deeper conventions, see [`.github/instructions/`](./.github/instructions/).

- **Reference-style links** for in-workshop pages and external docs:
  ```markdown
  See [Exercise 1][exercise-1] for context. The [Copilot CLI docs][cli-docs] explain.

  [exercise-1]: ../1-install-copilot-cli/
  [cli-docs]: https://docs.github.com/en/copilot/github-copilot-in-the-cli
  ```
  Define refs at the bottom of the page (or bottom of the partial — partials are self-contained).
- **Admonitions** use Starlight's `<Aside>` component:
  ```mdx
  <Aside type="note">Use `note`, `tip`, `caution`, or `danger`.</Aside>
  ```
  The Markdown renderer converts these to GitHub admonition syntax (`> [!NOTE]`).
- **No hard-wrapping** in repo-level Markdown files (READMEs, this file). Editors soft-wrap. Hard breaks are reserved for actual structural breaks.
- **Cross-repo links** (the demo app): always use `https://github.com/github-samples/tailspin-toys/...`. Don't link to files in *this* repo as if they were the template.

## CI and merge expectations

Every PR runs three checks (all must pass):

1. **`render-markdown.yml`** — Renders MDX → Markdown. Hard-fails on missing partials, unresolved component tags, broken internal links, or missing images.
2. **`pages.yml`** — Builds the Astro site. Must produce 29 pages.
3. **Lychee** — Link check on the rendered `workshop/`. 195 OK / 0 errors expected.

After merge to `main`:

- `pages.yml` deploys the updated site to GitHub Pages.
- `render-markdown.yml` regenerates `workshop/` and commits it back to `main` as `github-actions[bot]` with `[skip ci]` to avoid recursion.

You **don't** need to commit anything under `workshop/` in your PR. The bot owns it.

## Troubleshooting

- **"Module not found: `@astrojs/starlight/components`"** — run `npm install` inside `docs/`. Content lives under `docs/src/content/docs/`, so package resolution works automatically; no symlink hacks needed.
- **"No matching exports" or component renders as text** — check that the import in the host MDX page matches the component name and path exactly. Imports go at the top of the MDX file, after frontmatter.
- **Sidebar entry doesn't appear** — confirm you added it to `docs/astro.config.mjs` (it's manually maintained).
- **A lesson page renders but a partial inside it doesn't** — check that the partial's relative import path is correct (`../_partials/foo.mdx` from a path-folder lesson; adjust depth for `shared/<topic>/` pages).
- **Render script complains about a missing image** — confirm the image is in `docs/src/content/docs/images/` and the relative path matches.
- **Lychee reports a broken link** — most often a renamed lesson breaking a `[ref]: ../old-name/` definition. Update both the link target and any cross-page refs.
